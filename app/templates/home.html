01 / About

Not just
a developer.

I didn't start coding because of a bootcamp or a career pivot. I gravitated toward it — pulled by the feeling that building something functional from nothing is one of the few activities that rewards obsessive thinking.

What stuck wasn't just writing code. It was realizing that the way you structure a system matters more than the code inside it. I got hooked on architecture — on the idea that the right abstraction at the right level creates leverage you can't buy with more engineers.

I naturally think about extensibility, performance tradeoffs, and long-term structure — even in small side projects. I design them like production systems because the practice of building seriously is the only way to build well.

Off the keyboard, I treat everything like a system optimization problem — my workspace, my file structure, my trip planning. The obsessive organizing probably explains why I build structured tools for a living.

Currently seeking Summer 2026 internships in backend engineering, systems design, or developer tooling — ideally at companies that care about the lower layers: Oracle, the Linux Foundation, or similar.

Code Intelligence
Engine

2025
The gap no tool was filling

Most code quality tools do one of two things: lint surface-level style, or embed code into vectors and hope semantic similarity catches real problems. Neither understands architecture. I built a Rust CLI that combines AST traversal via Tree-sitter with semantic reasoning — detecting structural risks that exist between modules, not just inside them.

// Hardest decision

Static-only vs hybrid analysis. Choosing the indexing granularity that preserves structural context without blowing up memory. Early rules were too coupled to traversal logic — I redesigned around a clean rule abstraction layer so extension became additive, not surgical.

Portfolio CMS

2025
WordPress was overkill. Markdown was too raw.

I needed a place to exhibit work, not just display it. So I built a production-ready CMS from scratch: admin panel, blog engine, GitHub repo sync, security headers, and a cache layer. The goal wasn't to avoid existing tools — it was to understand every layer of the stack that usually gets hidden behind them. You're reading it right now.

// Hardest decision

GitHub's API rate limits mean you can't fetch repo data on every visit. I designed an internal cache with a manual refresh trigger — understanding the tradeoff between data freshness and API budget. Small problem, real production concern.

~80% fewer manual updates
70%+ test coverage
CSP+HSTS security headers

Kubernetes Docker Terraform Redis Java

CloudShrink

2024
Beginner cloud projects don't reflect real production

A URL shortener is a solved problem. I chose it deliberately — because the function is trivial, the architecture isn't. I built it as microservices on Kubernetes with Terraform-provisioned infrastructure to understand observability, scaling, and cost management without the distraction of a complex domain.

// Hardest decision

Monolith vs microservices. I chose microservices knowing it increased operational complexity — because that complexity was the point. Retrospectively, I'd collapse certain services in a real production startup. Architectural purity and operational pragmatism aren't always the same goal.
